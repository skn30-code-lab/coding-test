# 🚀 Coding Test Study

매일 꾸준히 문제를 풀고, 서로의 코드를 공유하며 함께 성장하는 코딩 테스트 스터디입니다.

---

## 📋 네이밍 규칙 한눈에 보기

| 항목 | 형식 | 예시 |
|------|------|------|
| 폴더명 | `yyMMdd_플랫폼_문제이름` | `260417_PMS_가장가까운같은글자` |
| 파일명 | `이름_문제이름.확장자` | `jexists_가장가까운같은글자.ipynb` |
| 브랜치명 | `feat/이름-yyMMdd` | `feat/jexists-260417` |
| 커밋명 | `feat: 문제이름` | `feat: 가장가까운같은글자` |

> 플랫폼: `BJ` (백준) / `PMS` (프로그래머스)

---

## ⏰ 운영 방식
- **모임 요일**: 월 / 수

- **시간**: 19:30 ~ 20:30 (최대 1시간)

- **진행 방식**
  - 정해진 문제를 각자 풀이합니다.
  - 풀지 못한 경우, 막힌 부분을 공유하고 함께 해결하는 시간을 갖습니다.
    - ("지금 이 부분이 잘 안 되는데, 도와주실 수 있을까요?")

- **불참 시**
  - 참여하지 못할 경우, 설명 주석이 달린 코드를 해당 날짜 폴더에 미리 업로드합니다.

문제별로 폴더를 생성하고, 해당 폴더 안에 본인의 파일을 업로드합니다.

### 1. 폴더명 규칙

  (예: `260402_BJ_설탕배달`)

- **프로그래머스**: `yyMMdd_PMS_문제이름`
  (예: `260402_PMS_입국심사`)

### 2. 파일명 규칙

```
이름_문제이름.py
이름_문제이름.ipynb
```

예시:
`jexists_설탕배달.py`

## 💬 커밋 컨벤션 (Commit Convention)

### 형식

```
feat: 문제이름
```

- 문제 하나당 커밋 하나로 합니다.
- 여러 문제를 풀었을 경우 각각 따로 커밋합니다.

### Prefix 의미 및 사용 상황

| Prefix   | 의미       | 사용 상황 |
|----------|------------|----------|
| feat     | 문제 풀이  | 문제를 완벽히 풀어서 제출할 때 |
| wip      | 진행 중    | 풀이 중인 코드를 중간 저장하거나 질문을 위해 올릴 때 |
| fix      | 수정       | 이미 올린 코드의 오류를 고치거나 로직을 수정했을 때 |
| refactor | 리팩토링   | 기능 변화 없이 코드를 효율적으로 개선했을 때 |
| docs     | 문서       | README 등 문서만 수정했을 때 |
| rename   | 이름 변경  | 파일명이나 폴더명을 규칙에 맞게 수정했을 때 |
| remove   | 삭제       | 불필요한 파일을 지울 때 |

## 🌿 브랜치 네이밍 규칙 (Branch Naming)

### 형식

```
feat/이름-yyMMdd
```

예시: `feat/jexists-260417`

### 규칙
- 브랜치는 세션(날짜)마다 새로 생성합니다.
- 이름: GitHub 사용자명 (소문자)
- 날짜: yyMMdd (6자리)
- 작업 완료 후 PR을 올리고 머지 시 브랜치를 삭제합니다.

## 🛠 참여 방법 (Workflow)

### 초기 설정 (최초 1회)

```bash
git config --global user.name "본인이름"
git config --global user.email "이메일주소"
git clone [Organization_URL]
```

### 데일리 루틴

```bash
# 1. 최신 상태 동기화
git pull origin main

# 2. 오늘의 브랜치 생성
git checkout -b feat/이름-yyMMdd
# 예: git checkout -b feat/jexists-260417

# 3. 문제 풀고 커밋 (문제 하나당 커밋 하나)
git add 파일명
git commit -m "feat: 문제이름"

# 여러 문제를 풀었을 경우 → 각각 커밋
git add 첫번째문제파일
git commit -m "feat: 가장가까운같은글자"
git add 두번째문제파일
git commit -m "feat: 유연근무제"

# 4. 브랜치 push
git push origin feat/이름-yyMMdd

# 5. GitHub에서 PR 생성 → 리뷰어 지정 → 머지 → 브랜치 삭제
```

## 👀 코드 리뷰 정책 (Code Review)

### 리뷰어 배정
- 매 세션마다 **1:1 페어 로테이션**으로 리뷰어를 배정합니다.
- 리뷰어는 상대방 PR에 코멘트를 남깁니다.
  - 더 나은 풀이나 다른 접근법이 있으면 공유해 주세요.
  - 잘 짠 코드도 칭찬 코멘트 환영입니다 :)

### 머지 조건
- 리뷰어가 Approve 하거나
- PR 올린 후 **48시간 내에 리뷰가 없으면 self-merge** 가능

### 브랜치 정리
- 머지 후 브랜치는 반드시 삭제합니다.

## 스터디 목적

이 스터디는 단순 문제풀이를 넘어서, 아래 목표를 가지고 진행됩니다.

→ 시험 대비를 통해 코딩테스트 기본기를 확실히 다집니다.

:three: 궁극적으로 좋은 개발자가 되는 것을 목표로 합니다.
→ 문제 해결력과 사고력을 키워 실무에서도 통하는 개발자로 성장합니다.
