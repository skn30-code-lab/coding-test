## ⚡ 빠른 시작

처음 참여하신다면 순서대로 따라하세요.

1. **[초기 설정](#초기-설정-최초-1회)** — git clone 등 최초 1회
2. **세션 시작 전** — 발제자가 GitHub Issue 및 폴더 생성
3. **문제 풀기** — 날짜 브랜치 생성 → 각자 파일 생성 → 커밋
4. **PR 올리기** — `Related to #1, #2` 작성 → 리뷰어 지정
5. **머지 후** — 브랜치 삭제, 모든 팀원 머지 완료 시 Issue close

## ⏰ 운영 방식

- **모임 요일**: 월 / 수
- **시간**: 19:30 ~ 20:30 (최대 1시간)

### 스터디 외
- 스터디 종료 후 → 다음날까지 발제자가 문제 확정 + Issue/폴더 생성
- 다음 스터디 전까지 → 각자 문제 풀기

### 스터디 시간
1. PR 올리기
2. 리뷰어 지정 + 리뷰 15분 + 머지
3. 발제자 발표 & 풀이 공유
4. 리뷰 중 공유할 내용 나누기
5. 다음 발제자 확정

- **불참 시**
  - 참여하지 못할 경우, 설명 주석이 달린 코드를 해당 날짜 폴더에 미리 업로드합니다.

## 🐛 GitHub Issues 활용법

> **발제자**가 세션 시작 전에 문제당 Issue 1개를 생성합니다.

### Issue 형식
제목:
```
[yyMMdd] 문제이름
```

본문:
```
- 일시:
- 레벨:
- 정답률:
- 링크:
```

예시:
- 제목: `[260420] 로또의최고순위와최저순위`
- 본문:
  ```
  - 일시: 4월 20일 월요일
  - 레벨: LV.1
  - 정답률: 60%
  - 링크: https://school.programmers.co.kr/learn/courses/30/lessons/77484
  ```

### PR과 연결하는 방법
PR 본문에 아래와 같이 작성합니다.
```
Related to #1, #2
```
> `Closes` 대신 `Related to` 를 사용합니다. `Closes`는 PR 머지 시 Issue가 자동으로 닫히기 때문에, 팀원 모두가 머지하기 전에 닫힐 수 있습니다.

### Issue close 시점
모든 팀원의 PR이 머지 완료되면 Issue를 **수동으로 close** 합니다.

## 📂 폴더 및 파일 규칙

### 1. 폴더명 규칙
> **발제자**가 세션 시작 전에 생성합니다.

- **백준**: `yyMMdd_BJ_문제이름`
  (예: `260420_BJ_설탕배달`)

- **프로그래머스**: `yyMMdd_PMS_문제이름`
  (예: `260420_PMS_로또의최고순위와최저순위`)

### 2. 파일명 규칙
> **각자** 본인 파일을 해당 폴더에 업로드합니다.

```
이름_문제이름.py
```

예시: `jexists_로또의최고순위와최저순위.py`

## 🌿 브랜치 네이밍 규칙 (Branch Naming)

### 형식

```
feat/이름-yyMMdd
```

예시: `feat/jexists-260420`

### 규칙
- 브랜치는 세션(날짜)마다 새로 생성합니다.
- 이름: GitHub 사용자명 (소문자)
- 날짜: yyMMdd (6자리)
- 작업 완료 후 PR을 올리고 머지 시 브랜치를 삭제합니다.

## 💬 커밋 컨벤션 (Commit Convention)

### 형식

```
feat: 문제이름
```

- 문제 하나당 커밋 하나로 합니다.
- 여러 문제를 풀었을 경우 각각 따로 커밋합니다.

### Prefix 의미 및 사용 상황

| Prefix   | 의미       | 사용 상황 |
|----------|------------|----------|
| feat     | 문제 풀이  | 문제를 완벽히 풀어서 제출할 때 |
| wip      | 진행 중    | 풀이 중인 코드를 중간 저장하거나 질문을 위해 올릴 때 |
| fix      | 수정       | 이미 올린 코드의 오류를 고치거나 로직을 수정했을 때 |
| refactor | 리팩토링   | 기능 변화 없이 코드를 효율적으로 개선했을 때 |
| docs     | 문서       | README 등 문서만 수정했을 때 |
| rename   | 이름 변경  | 파일명이나 폴더명을 규칙에 맞게 수정했을 때 |
| remove   | 삭제       | 불필요한 파일을 지울 때 |

## 🛠 참여 방법 (Workflow)

### 초기 설정 (최초 1회)

```bash
git config --global user.name "본인이름"
git config --global user.email "이메일주소"
git clone [Organization_URL]
```

### 데일리 루틴

```bash
# 0. 세션 시작 전 — 발제자가 Issue 및 폴더 생성
#    Issue: GitHub → Issues → New Issue → 제목: [260420] 문제이름
#    폴더: 260420_PMS_로또의최고순위와최저순위

# 1. 최신 상태 동기화
git pull origin main

# 2. 오늘의 브랜치 생성
git checkout -b feat/이름-yyMMdd
# 예: git checkout -b feat/jexists-260420

# 3. 문제 풀고 커밋 (문제 하나당 커밋 하나)
git add 파일명
git commit -m "feat: 문제이름"

# 여러 문제를 풀었을 경우 → 각각 커밋
git add 첫번째문제파일
git commit -m "feat: 로또의최고순위와최저순위"
git add 두번째문제파일
git commit -m "feat: 유연근무제"

# 4. 브랜치 push
git push origin feat/이름-yyMMdd

# 5. GitHub에서 PR 생성
#    본문에 Related to #1, #2 작성
#    → 리뷰어 지정 → 머지 → 브랜치 삭제
#    → 모든 팀원 머지 완료 시 Issue 수동 close
```

## 👀 코드 리뷰 정책 (Code Review)

### 리뷰어 배정
- 스터디 시간에 리뷰어를 지정합니다.

### 리뷰 방식
- 리뷰어는 상대방 PR에 코멘트를 남깁니다.
  - 더 나은 풀이나 다른 접근법이 있으면 공유해 주세요.
  - 잘 짠 코드도 칭찬 코멘트 환영합니다 :)

### 머지 조건
- 리뷰 후 스터디 당일에 머지합니다.

### 브랜치 정리
- 머지 후 브랜치는 반드시 삭제합니다.

## 스터디 목적

이 스터디는 단순 문제풀이를 넘어서, 아래 목표를 가지고 진행됩니다.

1️⃣ PCCE 취득을 목표로 합니다.
→ 시험 대비를 통해 코딩테스트 기본기를 확실히 다집니다.

2️⃣ 알고리즘에 대한 이해를 목표로 합니다.
→ 단순 암기가 아니라, "왜 이렇게 푸는지"를 이해하는 데 집중합니다.

3️⃣ 궁극적으로 좋은 개발자가 되는 것을 목표로 합니다.
→ 문제 해결력과 사고력을 키워 실무에서도 통하는 개발자로 성장합니다.
